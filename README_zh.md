# Org-GTD-Complete: 为 Emacs org-mode 设计的完整 GTD 工作流系统

[English](README.md) | [中文](README_zh.md)

本包在 Emacs org-mode 中完整实现了 David Allen 的《搞定》（GTD）方法论，提供了一套全面工作流系统。它涵盖 GTD 的五个核心步骤（收集、厘清、组织、检视、执行），并完整支持六层视野框架、自然计划模式、每周回顾流程、情境分类系统、两分钟原则以及所有标准 GTD 清单（项目、下一步行动、等待、将来/可能、参考资料）。

系统还包含收件箱处理、项目规划、定期回顾、参考资料管理、触发清单、头脑风暴工具和组织原则，帮助用户实现从日常任务到人生目标的垂直与水平关注管理。

## 安装

```elisp
M-x package-install org-gtd-complete
```

## GTD 理念

### 核心原则

GTD 方法论建立在几个根本性的哲学理念之上，这些理念构成了整个系统的基础：

1. **心智如水**：通过将任务外化到可信赖的系统中，让大脑从记忆负担中解放出来，专注于创造性思考而非存储信息。
2. **结果导向**：任何任务都必须有明确的期望结果，模糊的目标无法驱动有效行动，清晰的定义是执行的前提。
3. **系统信任**：建立一个完整、可靠的外部系统，让用户能够完全信任它，从而减少心理焦虑和决策疲劳。
4. **情境驱动**：行动的选择应基于当前的情境、可用时间、精力和工具，而非单纯的任务重要性或紧急性。
5. **持续进化**：通过定期检视和调整，系统需要不断适应变化的环境和目标，保持其相关性和有效性。

### 五个步骤

GTD 工作流包含五个连续步骤，形成一个闭环系统：

```
      +----------------+
      |      收集      |
      |    (Capture)   |
      +--------+-------+
               |
               v
      +--------+-------+
      |      处理      |
      |    (Clarify)   |
      +--------+-------+
               |
               v
      +--------+-------+
      |      组织      |
      |   (Organize)   |
      +--------+-------+
               |
               v
      +--------+-------+
      |      回顾      |
      |    (Reflect)   |
      +--------+-------+
               |
               v
      +--------+-------+
      |      执行      |
      |    (Engage)    |
      +----------------+
```

#### 1. 收集 (Capture)
将所有任务、想法、承诺和待办事项从大脑和外部来源收集到“收件箱”中。目标是清空大脑，确保没有遗漏。

#### 2. 处理 (Clarify)
对收件箱中的每个项目进行逐一处理：
- 问自己：“这是什么？”
- 如果不是可执行的任务，则丢弃、归档为参考资料或标记为“将来/可能”
- 如果是可执行任务，则确定下一步行动和期望结果

#### 3. 组织 (Organize)
将处理后的项目放入相应的系统中：
- **下一步行动清单**：可立即执行的具体行动
- **项目清单**：需要多个步骤才能完成的任务
- **等待清单**：委托他人或等待外部事件的任务
- **将来/可能清单**：暂时不执行但未来可能考虑的任务
- **参考资料**：需要保存但不需行动的信息

#### 4. 回顾 (Reflect)
定期检视系统以保持其更新和有效：
- **每日回顾**：检查当日行动清单和日程
- **每周回顾**：全面更新所有清单，回顾目标和项目进展
- **每月/年度回顾**：检视更高层次的目标和愿景

#### 5. 执行 (Engage)
基于当前情境、时间、精力和优先级，从清单中选择最合适的行动来执行。这是实际完成工作的阶段。

这五个步骤形成一个持续循环：执行后产生新的任务需要收集，定期回顾确保系统有效，从而持续提升个人生产力。

### 六层视野

GTD 的“六层视野”（Horizon of Focus）提供了一个从具体行动到人生目标的完整视角框架。每一层代表不同的关注层面，帮助用户在适当的抽象级别上进行思考和规划。

关键规则：
- **L4-L6 只设置在项目或职责范围层面**：高层视野（目标、愿景、宗旨）不应直接关联到单个行动，而是通过项目或职责范围来体现。
- **单个行动最多关联到 L3（Area）**：具体行动可以属于某个责任领域，但不能直接拥有更高层次的视野标签。
- **行动通过所属项目继承 L4-L6**：当行动属于一个项目时，它自动继承该项目所关联的高层视野，无需重复设置。
- **单独行动（不属于任何项目）不应拥有 L4-L6 视野**：这些行动只应关联到 L3 责任领域，不应触及更高层次。

以下是六层视野的基本层级结构：

```
  +---------------------------+
  |    L6: 宗旨与核心价值     |
  |  (Purpose & Core Values)  |
  +-------------+-------------+
                ^
                |
  +----------------+----------+
  |          L5: 愿景         |
  |          (Vision)         |
  +-------------+-------------+
                ^
                |
  +-------------+-------------+
  |          L4: 目标         |
  |          (Goals)          |
  +-------------+-------------+
                ^                    ◉ 就此终止
                |                     |
  +-------------+---------------------+-----+
  |              L3: 责任领域               |
  |        (Areas of Responsibility)        |
  +-------------+---------------------+-----+
                ^                     ^
                |                     |
  +-------------+-------------+       |
  |          L2: 项目         |  不属于任何项目
  |         (Projects)        |       |
  +-------------+-------------+       |
                ^                     |
                |                     |
  +-------------+---------------------+-----+
  |             L1: 下一步行动              |
  |             (Next Actions)              |
  +-----------------------------------------+
```

关键洞察：
- L4-L6 只能通过 L2 项目继承
- 无项目的 L1 行动最高只能关联到 L3（责任领域）
- L1 始终是 L1，区别在于连接点
```

（注：图中使用两条不同的路径和标注来直观展示两种行动的区别：项目行动通过实线箭头向上连接到 L6，独立行动通过虚线箭头仅连接到 L3 并在该层终止。）

#### 各层详细说明

**L1: 下一步行动 (Next Actions)**
- 可立即执行的具体物理行动。
- 例如：“给客户发邮件确认会议时间”、“购买打印机墨水”。
- **关键区分**：
  - 属于项目的行动：继承该项目的 L2-L6 视野
  - 单独行动：最多关联到 L3（责任领域），不涉及 L4-L6

**L2: 项目 (Projects)**
- 需要多个步骤才能完成的期望结果。
- 每个项目应关联到至少一个责任领域（L3）。
- 例如：“完成季度报告”、“组织团队建设活动”。
- 项目本身可以设置 L4-L6 标签。

**L3: 责任领域 (Areas of Responsibility)**
- 生活中需要持续关注和维护的重要领域。
- 例如：健康、家庭、职业发展、财务管理。
- 所有行动（无论是项目内还是单独）都可以关联到责任领域。

**L4: 目标 (Goals)**
- 未来1-2年内希望实现的中期目标。
- 例如：“晋升为技术主管”、“完成马拉松训练”。
- 目标通过项目来实现，不应直接关联到单个行动。

**L5: 愿景 (Vision)**
- 未来3-5年的长期展望和生活图景。
- 例如：“成为领域专家”、“建立稳定的被动收入”。
- 愿景通过目标来具体化，进而通过项目实现。

**L6: 宗旨与核心价值 (Purpose & Core Values)**
- 人生的根本意义和指导原则。
- 例如：“帮助他人成长”、“追求科技创新”。
- 这是最高层次的视野，指导所有下层决策。

#### 实践示例

假设你的核心价值（L6）是“保持健康生活”，愿景（L5）是“拥有充沛精力”，目标（L4）是“今年完成半程马拉松”。为实现这一目标，你设立一个项目（L2）“半程马拉松训练计划”，该项目属于责任领域（L3）“健康”。

**项目行动**：
- 项目内的下一步行动（L1）：如“每周三次跑步训练”、“购买专业跑鞋”
- 这些行动**自动继承**项目的 L2-L6 视野

**单独行动**：
- 不属任何项目的行动：如“今天称体重”
- 可以关联到责任领域（L3）“健康”
- **但不应**关联到 L4（目标：“完成半程马拉松”）、L5（愿景）或 L6（核心价值）

在 org-gtd-complete 中，你只需在项目层面设置 L4-L6 的标签，所有属于该项目的行动将自动继承这些高层视野。单独行动则只需标记责任领域（L3），系统会确保它们不会错误关联到 L4-L6。这确保了系统的简洁性和一致性。

### 清单系统

GTD 的清单体系采用 **实体清单** 与 **虚拟清单** 相结合的层次结构，确保任务既能按组织结构完整存储，又能按执行状态灵活取用。

#### 关键概念区分

在理解 GTD 清单系统时，需要明确几个关键概念的区别：

**下一步行动 (Next Actions) vs 单一行动 (Single Actions)**

- **下一步行动 (Next Actions)**：指**当前可以立即执行**的具体行动。这是一个**虚拟清单**，其中的行动来自两个来源：
  1. 项目清单中每个项目的**下一个具体步骤**
  2. 单一行动清单中**当前可执行的行动**

- **单一行动 (Single Actions)**：指**不属于任何项目**的独立可执行行动。这是一个**实体清单**，存储所有不需要多步骤完成的独立任务。

**简单来说**：
- 单一行动清单是**存储**独立任务的地方
- 下一步行动清单是**动态生成**的视图，包含所有当前可执行的任务（既包括项目的下一步，也包括可执行的单一行动）

**等待清单 (Waiting For List)**：指**已委托他人或等待外部事件**的任务。这也是一个**虚拟清单**，其中的任务同样来自项目清单或单一行动清单。

#### 核心实体清单

**实体清单** 是存储任务原始记录的清单，它们是 GTD 系统的基石：

1. **项目清单 (Projects List)**
   记录所有需要多个步骤才能完成的期望结果。每个项目包含其完整计划（所有步骤）、预期成果和关联的高层视野（L4-L6）。

2. **单一行动清单 (Single Actions List)**
   记录不属于任何项目的独立可执行行动。这些行动直接与责任领域（L3）关联，不涉及项目级规划。

#### 向上映射的虚拟清单

**虚拟清单** 是根据不同视角从实体清单动态生成的视图，不重复存储任务：

- **L3 责任领域 (Areas of Responsibility)**
  映射实体清单中所有行动所属的生活领域（如健康、职业、家庭等）。

- **L4 目标 (Goals)**
  映射项目清单中为实现特定目标而设立的项目。

- **L5 愿景 (Vision)**
  映射与长期愿景相关的项目集合。

- **L6 宗旨与核心价值 (Purpose & Core Values)**
  映射反映人生核心价值与意义的所有项目。

向上映射遵循六层视野规则：**L4-L6 仅通过项目继承**；**单一行动最高映射到 L3**。

#### 向下映射的虚拟清单

基于执行状态，实体清单中的行动可动态归入以下虚拟清单：

1. **下一步行动清单 (Next Actions List)**
   包含所有 **可立即执行** 的行动，这些行动来自：
   - 项目清单中当前项目的 **下一个具体步骤**
   - 单一行动清单中 **当前可执行的行动**

2. **等待清单 (Waiting For List)**
   包含所有 **已委托他人或等待外部事件** 的行动，这些行动同样来自项目清单或单一行动清单。

向下映射遵循 **中转站原则**：当项目或单一行动中的某个步骤需要执行时，它从实体清单临时“移出”到相应的虚拟清单（下一步行动或等待清单），在实体清单中保留占位符（如“进行中”、“等待中”状态）。完成后再回归实体清单并标记为完成。

#### 辅助清单

- **收件箱 (Inbox)**：临时收集所有未处理的输入。
- **将来/可能清单 (Someday/Maybe List)**：暂不执行但未来可能考虑的想法。
- **参考资料 (Reference Material)**：不需行动但需保留的信息。

#### 清单互动示意图

```
            +-------------------------------------+
            |         实体清单 (实体存储)         |
            |  +-------------------------------+  |
            |  |      项目清单 (Projects)      |  |
            |  +-------------------------------+  |
            |  | 单一行动清单 (Single Actions) |  |
            |  +-------------------------------+  |
            +------------------+------------------+
                               |
             +-----------------+-----------------+
             |                                   |
             v                                   v
      +------+-------+                   +-------+------+
      | 向上虚拟清单 |                   | 向下虚拟清单 |
      | (六层视野)   |                   |  (执行状态)  |
      +------+-------+                   +-------+------+
             |                                   |
   +-----+---+-+------+                   +------+------+
   |     |     |      |                   |             |
   v     v     v      v                   v             v
  L3    L4    L5     L6               下一步行动        等待
(领域)(目标)(愿景)(核心价值)        (Next Actions)  (Waiting For)
```

#### 实践意义

1. **完整性**：实体清单保证所有任务（项目步骤、独立行动）都有完整记录，便于整体检视。
2. **灵活性**：虚拟清单让用户只需关注当前相关视图（如“今天在办公室能做什么”、“所有等待中的项目任务”）。
3. **一致性**：任务状态变更（委托、执行、完成）仅在实体清单中记录一次，虚拟清单自动同步。

所有清单应通过 **定期回顾**（尤其是每周回顾）保持时效，确保系统持续可靠。
